# Reflection

## 1. What was the most technically challenging part of this project?
The most technically challenging aspect was designing a deterministic rule engine for the AI tool audit that could handle messy, unpredictable user inputs while still providing mathematically accurate savings calculations. Initially, I considered offloading the entire analysis to an LLM, but I quickly realized that LLMs are prone to hallucinations when calculating specific dollar amounts and market rate comparisons. Building the `audit.service.ts` engine required careful consideration of edge cases—such as users entering duplicate tools under different plans, or selecting an enterprise plan for a 2-person team. Ensuring the overlap rules (e.g., comparing Cursor and GitHub Copilot) aggregated spend accurately across all identical tool entries before recommending cancellations was surprisingly complex. The deduplication logic also required refinement to ensure valid recommendations weren't arbitrarily dropped. Ultimately, building a strict, TypeScript-enforced deterministic engine that feeds a raw JSON summary to the LLM provided the perfect balance of accuracy and conversational readability.

## 2. If you had to scale this to 10,000 audits per day, what architectural changes would you make?
Handling 10,000 audits per day would immediately expose bottlenecks in the synchronous LLM calls and database writes. Currently, the client waits for the Express backend to run the rules, call the Anthropic API, save to Neon Postgres, and return the `shareId`. At high volume, this would lead to timeout errors and rate-limiting. I would decouple the process by introducing an asynchronous message queue (like AWS SQS or Redis BullMQ). The API would instantly return a `jobId` and save the raw input to the database. A pool of background workers would process the LLM calls and update the database record upon completion. The frontend would then use WebSockets or short-polling to display the final result. Additionally, I would implement Redis caching for the benchmark data queries, as computing percentiles across thousands of rows in real-time would become computationally expensive.

## 3. What is one trade-off you made due to time constraints, and how would you fix it?
A significant trade-off was the decision to duplicate the pricing logic and constants across both the Next.js client (`client/lib/pricing.ts`) and the Node.js backend. I did this to provide instant, real-time recalculations on the frontend UI without making network round-trips for every form change, while still ensuring the backend could independently verify the data securely. However, this violates the DRY (Don't Repeat Yourself) principle and creates a risk of schema drift if a tool's price changes and only one file is updated. To fix this properly, I would transition the project into a true monorepo using Turborepo or npm workspaces. I would extract the pricing catalog, TypeScript interfaces, and shared utility functions into a shared `@ai-auditor/core` package that both the frontend and backend import. This would guarantee total synchronization.

## 4. How did you balance user experience with data accuracy in the form design?
Balancing UX and data accuracy is a tightrope. I wanted the form to feel frictionless so users wouldn't abandon it, but I needed highly accurate data to generate meaningful audit results. I achieved this by using local state batching and instant visual feedback. For example, if a user inputs "1" seat for a "Team" plan, the UI subtly updates the estimated spend based on the official market rate but still allows them to manually override the cost if they have a custom negotiated discount. To prevent data loss, I implemented local storage persistence (`localStorage`) so the form saves on every keystroke. This means if a user accidentally closes the tab while looking up their API spend, they don't lose their progress. Finally, I added client-side bounds checking and a hidden honeypot field to sanitize inputs and prevent bot spam without subjecting real users to annoying CAPTCHAs.

## 5. What did you learn about the AI tooling market while researching for this app?
Researching the pricing models of various AI tools revealed how fragmented and opaque the market is becoming. Many companies default to purchasing ChatGPT Team or Enterprise licenses for their entire staff, completely unaware that their developers are simultaneously expensing GitHub Copilot and Cursor individually. The overlap in functionality is massive. I also learned that "per-seat" billing models for AI tools are fundamentally misaligned with actual usage for non-technical teams, resulting in massive shelfware. Most surprisingly, the tier jumps (e.g., from Claude Pro to Claude Team) often introduce arbitrary seat minimums that actively penalize small startups. This research validated the core hypothesis of the app: companies don't need more AI tools; they need to consolidate and optimize the ones they are already paying for.
